TESTING GUIDE - MEDICONNECT
===========================

1. DOCUMENT PURPOSE
-------------------
This document guides QA engineers, developers, and testers on how to test the MediConnect system from three perspectives:
- Environment and installation testing
- Functional testing by module
- Regression testing after code changes

The guide is written so that new team members can read it, test the system, and continue adding new cases later.

2. TEST SCOPE
-------------
The following functional areas should be tested:
- Public pages
- Authentication
- User profile
- Doctor verification workflow
- Doctor list and booking
- Appointment lifecycle
- Prescription
- Blog
- Admin management modules
- File uploads
- Validation, permissions, and edge cases

Not currently in high-priority scope:
- Deep performance benchmarking
- Professional security penetration testing
- API testing, because the project currently does not expose a separate API layer

3. CURRENT AUTOMATION STATUS
----------------------------
The tests/ directory currently contains only:
- tests/Feature/ExampleTest.php
- tests/Unit/ExampleTest.php

That means:
- There is no real automated test suite yet for booking, approval, prescription, or admin CRUD flows.
- Manual testing and regression checklists are currently the main quality strategy.
- The team should prioritize adding Feature tests based on the recommendations in Section 14.

4. TEST ENVIRONMENT PREPARATION
-------------------------------
4.1. Requirements
- PHP 8.3+
- Composer
- Node.js + npm
- MySQL or SQLite
- Important PHP extensions: mbstring, dom, xml, xmlwriter, json, tokenizer, libxml

4.2. Setup steps
- composer install
- npm install
- copy .env.example .env
- php artisan key:generate
- configure DB settings in .env
- php artisan migrate --seed
- npm run build or npm run dev
- php artisan serve

4.3. Notes about the analyzed zip snapshot
In the environment used to inspect the uploaded zip, php artisan route:list and php artisan test could not run because some PHP extensions were missing, especially:
- mbstring
- dom
- xml
- xmlwriter

This is an environment/runtime issue, not proof that the application business logic is broken. For proper testing, install the required extensions first.

11. FILE UPLOAD TESTING
-----------------------
Must be tested for:
- avatar
- citizen_id_front
- citizen_id_back
- degree_image
- blog thumbnail

General checklist:
- valid jpg/jpeg/png/webp uploads
- file > 2MB is blocked
- non-image file is blocked
- successfully uploaded file exists at the expected path
- old file is removed when replaced, where applicable

12. EDGE CASES THAT NEED EXTRA ATTENTION
----------------------------------------
- Booking on the current day for a time that has already passed
- Booking on the last valid day of the 7-day booking window
- Schedule with start_date = end_date
- Empty max_patients input -> confirm whether the system defaults to 10 when creating a schedule
- User with an existing pending doctor profile submits verify-doctor again
- Admin approves/rejects the same doctor multiple times in sequence
- Deleting medication/medicine type with existing dependencies
- Switching role user <-> admin in patient management
- Duplicate blog slug
- Strange characters in thumbnail/blog/avatar file names

16. RELEASE TEST COMPLETION CRITERIA
------------------------------------
A release can be considered ready when:
- All P1 tests pass
- No open Critical/High bugs remain
- Permission tests pass
- Upload tests pass
- Admin doctor approval flow passes
- Booking -> prescription -> review flow passes
- Relevant regression suite passes

17. CONCLUSION
--------------
MediConnect is currently suitable for structured manual testing, but its automation coverage is still very low. This Testing Guide prioritizes the most critical business flows in the product: login, appointment booking, doctor approval, prescription, and administration. The team should use this document as a baseline and continue expanding test cases and automation as the system stabilizes.
